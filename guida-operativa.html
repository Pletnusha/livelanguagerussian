1. Come funziona il sistema

Flusso A: Registrazione gratuita

  1. L'utente va su /register.html (o /auth.html?mode=register) e compila il form
  2. Il sistema crea l'account in Supabase Auth, scrive il profilo nella tabella profiles e chiama il webhook n8n
  3. L'utente riceve una email automatica con username, password e link diretto agli esercizi
  4. Può accedere agli esercizi con data-access="public"
  5. Dopo il login viene reindirizzato a /livello_elementare/ (o alla pagina di origine se presente il parametro ?redirect=)

Flusso B: Abbonamento (Gruppo / Light / Base)

  1. L'utente va su /courses.html e sceglie il piano
  2. Compila il form di registrazione su /register.html?plan=gruppo (o light / base)
  3. Viene reindirizzato su Stripe per il pagamento (con client_reference_id = user_id)
  4. Stripe notifica il backend → n8n aggiorna exercisepaid = true e registration_status = paid
  5. L'utente riceve email con le credenziali e accesso completo agli esercizi data-access="paid"

3. Creare nuovi esercizi

Struttura delle cartelle

Gli esercizi seguono questa struttura:

/livello/argomento/nome-esercizio/index.html

Esempi esistenti:

  • /principianti/casi/genitivo-di-chi-e/index.html
  • /livello_elementare/verbi-di-moto/index.html

Passaggi per aggiungere un nuovo esercizio

1. Crea la cartella nella posizione giusta, es. /principianti/casi/nuovo-argomento/
2. Copia un index.html esistente dalla stessa sezione come punto di partenza
3. Modifica titolo, testo introduttivo e contenuti dell'esercizio
4. Imposta il livello di accesso su ogni pannello esercizio (vedi sotto)
5. Assegna ID univoci a ogni pannello ed esercizio
6. Aggiungi il link nella pagina indice del livello (/principianti/index.html ecc.)

Livelli di accesso

Ogni pannello esercizio ha un attributo data-access che determina chi può vederlo:

data-accessChi può vederloCorrisponde a
publicTutti gli utenti registrati (gratuiti e paganti)Utente loggato = "studente"
paidSolo utenti con abbonamento (exercisepaid = true)Abbonato
Nota: il livello student è definito nel codice ma non attualmente attivo. Per semplicità, "studente" = qualsiasi utente loggato = data-access="public". I pannelli marcati data-access="student" negli scaffold del passato vanno considerati equivalenti a public finché non viene attivato il livello student nel backend.

Esempio nel codice HTML:

<!-- Esercizio gratuito -->
<div class="exercise-panel" id="panel-nome-1" data-access="public" hidden>
    ...
</div>

<!-- Esercizio solo abbonati -->
<div class="exercise-panel" id="panel-nome-2" data-access="paid" hidden>
    ...
</div>
Importante: ogni pannello deve avere un id univoco in tutto il sito. Usa una convenzione chiara, es. panel-genitivo2-01, panel-genitivo2-02, ecc.

4. Pagine di tipo Guida (contenuto teorico)

Oltre alle pagine di esercizi interattivi, esistono pagine di guida teorica con tabelle di riferimento. Seguono la stessa struttura pannello-card ma il tipo è Guida e il contenuto è testuale.

Esempio: ИДТИ — Usi speciali

/livello_elementare/verbi-di-moto/idti-usi-speciali/
├── index.html     ← 7 pannelli collassabili (meteo, tempo, processi...)
└── exercises.js   ← solo initPanelManager({ enableAccessControl: true })

Tutti i pannelli usano data-access="public" (visibili a qualsiasi utente loggato). Il tipo mostrato nella card è Guida anziché Drag & Drop / Flashcard.

Struttura CSS aggiuntiva: le pagine guida aggiungono un <style> inline con le classi .guida-table, .guida-regola-oro e .guida-nota — non fanno parte di main.css e sono locali alla pagina.

6. Funzionalità per il Copywriter

6.1 Spiegazione dell'errore negli esercizi Drag & Drop

Ogni esercizio nell'array di un DragDropExercise accetta un campo opzionale explanation. Se presente, viene mostrato all'utente quando risponde in modo errato, con la forma corretta e la spiegazione grammaticale.

Comportamento del feedback:

SituazioneMessaggio visualizzato
Risposta corretta✓ Corretto!
Risposta errata con explanation✗ [forma corretta] + spiegazione sulla riga sotto
Risposta errata senza explanationRisultato: X/Y (comportamento invariato)
Nota: il campo è completamente opzionale. Gli esercizi già esistenti senza explanation continuano a funzionare esattamente come prima.

Formato dell'oggetto esercizio con il nuovo campo:

{
  instruction: "Completa la frase scegliendo la forma corretta",
  text: "Я иду {{1}}, нужно купить хлеб.",
  words: ["магазине", "магазина", "магазин"],
  correctAnswers: { 1: "магазин" },
  explanation: "в + accusativo → магазин (movimento verso un luogo)"
}

La spiegazione appare sotto la riga ✗ магазин, nello stesso stile visivo rosso del feedback di errore. Non è necessario modificare nessun file HTML.

6.2 CTA dopo il Panel 2 (blocco promozionale inline)

È possibile mostrare un blocco CTA (call to action) tra la card del Panel 2 e la card del Panel 3, solo dopo che l'utente ha aperto il Panel 2. Appare una sola volta per sessione.

La CTA si configura passando il parametro ctaAfterPanel2 a initPanelManager nel file exercises.js della pagina. Il Copywriter fornisce il testo HTML; il Tech Specialist aggiorna il file.

Struttura del parametro:

ctaAfterPanel2: {
  panelId: 'panel-mal-open-2',   // ID del pannello Panel 2 di questa pagina
  html:    '...'                 // contenuto HTML della CTA (testo del Copywriter)
}

Esempio completo in exercises.js:

initPanelManager({
  initializers: {
    'panel-mal-open-1': () => new DragDropExercise({ ... }),
    'panel-mal-open-2': () => new FlashcardExercise({ ... }),
    // ...
  },
  enableAccessControl: true,
  ctaAfterPanel2: {
    panelId: 'panel-mal-open-2',
    html: `
      <h2 class="cta-title">Vuoi sbloccare tutti gli esercizi?</h2>
      <p class="cta-text">Testo fornito dal Copywriter per questa sezione.</p>
      <a href="/courses.html" class="cta-button">Scopri il corso</a>
    `
  }
});

Regole operative:

  • Il panelId deve corrispondere all'id esatto del pannello Panel 2 di quella pagina
  • Il parametro ctaAfterPanel2 è opzionale — se omesso, nessuna CTA viene iniettata
  • Nessuna modifica ai file HTML è necessaria: il blocco viene inserito nel DOM in automatico
  • La sessione viene tracciata separatamente per ogni pagina (una CTA per pagina per sessione)
Workflow consigliato: il Copywriter scrive i testi CTA per ogni sezione e li consegna al Tech Specialist, che aggiorna i rispettivi file exercises.js aggiungendo il parametro ctaAfterPanel2.

7. Pubblicare le modifiche (deploy)

Il sito è su GitHub (Pletnusha/livelanguagerussian) e viene servito staticamente. Il processo è:

1. Verifica lo stato locale con git status
2. Se ci sono branch divergenti (es. dopo modifiche remote), allinea con git pull --rebase origin main
3. Aggiungi i file modificati: git add <file> (evita git add -A per non includere file sensibili)
4. Crea il commit: git commit -m "Descrizione delle modifiche"
5. Pubblica: git push origin main
Attenzione: non fare mai git push --force su main. Se il push viene rifiutato, fare prima git pull --rebase origin main e poi riprovare.

Anteprima locale

Per vedere le modifiche in locale prima di pubblicarle:

cd /percorso/del/sito
python3 -m http.server 8080

Poi aprire http://localhost:8080 nel browser. Il controllo accessi Supabase non funzionerà in locale (tutti i pannelli saranno visibili), ma la struttura e il layout sono verificabili.
